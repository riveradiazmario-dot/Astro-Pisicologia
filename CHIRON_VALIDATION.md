CHIRON VALIDATION
=================

Objective
---------
Compare the repository's current Chiron (Quirón) implementation (simple Kepler-based longitude) against a higher-fidelity two-body orbital-element model across 100 dates (1900–2100), measure differences, and evaluate suitability for professional use. Attempts to fetch Swiss Ephemeris / Astro.com / AstroSeek reference data were started but require dedicated ephemeris integration (see Limitations).

Method
------
- Implemented the library's current algorithm (engine_simple) reproduced from src/astronomy/engine.ts.
- Implemented a more complete orbital-element heliocentric ecliptic longitude (orbital_model) using published osculating elements (Wikipedia/JPL) at epoch 2021-07-01: a, e, i, Ω, ω, M0 and the period from literature.
- Generated 100 evenly spaced test dates between 1900-01-01 and 2100-12-31.
- For each date computed engine_simple longitude and orbital_model heliocentric ecliptic longitude; computed shortest angular difference (degrees).
- Wrote results to /tmp/chiron_validation.csv (CSV included in analysis artifact, not committed).
- Attempted a best-effort query to JPL Horizons for a few sample dates; the API returned an input-format error for the QUANTITIES parameter (requires different formatting) — see "Limitations and next steps".

Results
-------
- Tests run: 100 dates (evenly spaced).
- Max absolute difference: 42.347888°
- Min absolute difference: 24.358304°
- Mean absolute difference: 29.324456°

Observations
------------
- The engine's getChironLongitude implementation in src/astronomy/engine.ts computes longitude as `perihelionLongitude + trueAnomaly` derived from a mean-motion Kepler solution. This ignores orbital inclination, longitude of ascending node, and the full 3D geometry of the orbit, and therefore produces only a very coarse angular approximation.
- The orbital-element model (used here) includes inclination, node and argument of perihelion; differences vs the engine are large (24°–42°), consistent across the century span tested.

Limitations
-----------
- This validation did NOT compare directly to Swiss Ephemeris, Astro.com or AstroSeek authoritative outputs for all 100 dates due to environment constraints and the need for a proper ephemeris client (swisseph/native binding) or correct JPL Horizons request parameters. A best-effort JPL Horizons call was attempted and returned a QUANTITIES syntax error; the API is reachable and should be used with corrected parameters (or JSON mode) in a follow-up step.
- The orbital-element model here uses osculating elements from a public source (Wikipedia/JPL summary) and a two-body solution. It is an improved model vs the engine's oversimplification but still lacks planetary perturbations and full ephemeris corrections (TT/UT, ΔT) that Swiss Ephemeris/JPL/Horizons include.

Conclusion and recommendation
-----------------------------
- Professional astrological use demands sub-degree accuracy for small bodies like Chiron (target ≤0.1°–0.5° for publication-grade charts; conservative threshold ≤1° for many applications).
- The current implementation (simple Kepler + perihelionLongitude shortcut) shows average differences ≈29° vs a modest orbital-element model and is therefore NOT acceptable for professional usage.

Next steps (recommended)
------------------------
1. Integrate Swiss Ephemeris (libse) or the swisseph npm binding, or use JPL Horizons REST correctly, to obtain authoritative Chiron positions. Produce a canonical reference table (daily or hourly as needed) and cache results.
2. Add automated regression tests comparing the code's Chiron output to Swiss Ephemeris/JPL for a representative sample (100+ dates) with acceptance thresholds (e.g., ≤0.5°).
3. Replace getChironLongitude with calls to an ephemeris-backed implementation (or use astronomy-engine if it supports small-body ephemerides) and re-run validation.
4. Fix retrograde detection for small bodies by computing instantaneous speed from ephemeris instead of ±24h finite differences.
